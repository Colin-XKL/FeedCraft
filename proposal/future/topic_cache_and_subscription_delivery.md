# 领域三：Topic 缓存与订阅交付

> 状态：未开始

## 1. 背景

按照 Topic 的产品目标，用户访问 `/topic/:id` 时，不只是要“能执行”，还要“能稳定订阅”。

Topic 和 Recipe 不同：

- Recipe 通常是单源处理
- Topic 是多源并发聚合，耗时和失败面都更大

如果每次请求都实时全量聚合：

- 首次访问会慢
- 高并发访问会重复打上游
- 多个上游波动时，Topic 的订阅稳定性会明显下降

因此，Topic 需要一层专门的订阅交付策略，而不是只依赖引擎能跑通。

## 2. 需求场景

典型用户会把 Topic 链接直接丢给 RSS Reader 周期轮询。

这意味着服务端需要兼顾：

- 冷启动时可以生成结果
- 热路径下可以快速返回
- 上游部分波动时仍尽量保证订阅可用

典型期望：

- 第一次访问时允许实时聚合
- 后续在 TTL 内优先命中缓存
- 缓存过期后重新生成快照
- 未来如有需要，可演进到后台预热或更精细的缓存控制

## 3. 当前进展

- `proposal/completed/topic_aggregation_design.md` 已经明确了 Topic TTL 缓存方向
- Redis 基础设施在项目中已经存在，见 `internal/util/cache.go`
- 系统已有公开 RSS 返回路径，说明订阅输出模式是成熟的

当前缺口：

- 还没有 Topic 专用缓存实现
- 还没有 Topic 的 TTL 策略常量或缓存 key 规范
- 还没有 Topic 的订阅级别端到端交付测试
- 还没有明确“失败时是否回退旧缓存”的策略

## 4. 未来待办

### 4.1 实现 Topic TTL 缓存

- 为 Topic 引入独立缓存 key 设计
- 缓存内容建议直接存最终 RSS XML 或最终 `CraftFeed` 的可序列化快照
- 先采用简单固定 TTL，和 proposal 保持一致

### 4.2 明确缓存命中与刷新逻辑

- 请求 `/topic/:id` 时：
  - 命中有效缓存 -> 直接返回
  - 未命中或已过期 -> 同步执行聚合并刷新缓存
- 保持实现简单，先不做复杂后台刷新

### 4.3 定义失败场景下的交付策略

- 若实时聚合失败，是否允许回退到旧缓存
- 若部分成功，是否直接交付部分结果
- 若全部失败，是否返回错误还是返回最近快照

首版建议先做最简单且行为清晰的版本：

- 有有效缓存优先返回缓存
- 没缓存且执行失败则返回错误

### 4.4 增加性能与交付测试

- 缓存命中测试
- 缓存失效后重建测试
- 并发访问下的行为测试
- 上游部分失败时的交付测试

## 5. 追踪建议

这个领域完成的标准是：

- Topic 对 RSS Reader 来说是一个稳定可轮询的订阅地址
- 常规访问路径不需要每次都实时全量聚合
- 缓存与错误时的交付行为是明确、可预测、可测试的
