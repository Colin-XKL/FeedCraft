# 高级去重策略 (Advanced Deduplication)

## 1. 需求场景
在 RSS 聚合（Topic Aggregation）场景中，一个 Topic 往往关联多个数据源（Recipe）。多源抓取极易产生重复内容。例如：
* 同一篇官方新闻稿被多个不同的科技媒体转载。
* 不同的作者对同一热点事件进行了报道（语义高度重合）。
如果仅仅依靠 URL 或者文章标题的“精确匹配（Exact Match）”去重，会导致用户订阅的 Topic 流中充斥着大量冗余信息，严重影响阅读体验。因此，系统需要能够识别并过滤掉“实质内容相同或高度相似”的文章。

## 2. 当前已有的一些想法
系统在未来可以引入多级防御的智能去重引擎：
* **基础去重 (Level 1)**: 基于 URL、GUID 或标题 Hash 的精确去重（MVP 阶段已实现）。
* **浅层相似度 (Level 2)**: 基于标题和摘要的文本相似度计算，例如计算编辑距离（Levenshtein）或 Jaccard 相似度。该方案不需要调用大模型，计算成本低，能快速过滤掉大量跨站的“通稿”。
* **语义/LLM 去重 (Level 3)**:
  * **向量化方案 (Embedding)**: 将抓取到的文章内容转换为向量，存储到轻量级 VectorDB（如基于 SQLite 的 VSS 或 Chroma）中。新文章入库前计算余弦相似度，超过特定阈值判定为重复。
  * **大模型判别方案 (LLM Prompting)**: 当两篇文章在浅层判断中属于“疑似重复”区间时，将其发送给 LLM，通过特定的 Prompt 让模型判定它们是否属于同一报道的简单洗稿。
* **持久化指纹库 (Fingerprint DB)**: 高级去重需要跨越不同的抓取批次和时间线。因此需要一张持久化的 `ArticleFingerprint` 数据库表，保存历史文章的特征（Hash/Embedding）。此外，为了避免数据无限膨胀，指纹库需要配合 TTL（如保留 30 天的特征数据）进行定期清理。

## 3. 与当前架构关联的地方
* **去重模块抽象**: 在现有的 Topic 处理流水线中，需预留出一个 `Deduplicator` 接口。当前 MVP 是该接口的简单内存实现（基于 URL Hash），未来可替换为注入了 DB/Redis 及 LLM Client 的复杂实现。
* **数据结构扩展**: 内部流通的数据模型 `CraftItem` 需要能够承载相关的特征属性（例如新增 `Fingerprint` 或 `Embedding` 字段）。
* **执行阶段**: 会作为 `TopicCrafts` 全局处理流中的关键一环，在各个单源 Recipe 并发抓取并合并后、最终排序截断前触发。
