# 领域四：Topic 管理体验与运行可观测

> 状态：已完成

## 1. 背景

TopicFeed 不是一个一次性功能，而是一类长期运行的聚合节点。用户不仅需要创建它，还需要长期维护它、排查它、理解它当前是否健康。

因此 Topic 的“可用性”不只取决于引擎和路由，还取决于：

- 后台是否容易配置
- 配置错误是否容易发现
- 运行异常是否容易定位

## 2. 需求场景

典型使用场景包括：

- 用户配置多个 `input_uris`，希望快速发现 URI 是否写错
- 用户配置聚合步骤，希望知道当前配置是否合法
- 某个上游 Recipe 持续失败时，希望从后台看到 Topic 已部分降级
- 用户希望知道 Topic 为什么没有产出内容，是全部上游失败、去重后为空，还是 limit 过严

## 3. 当前进展

- Admin 侧 Topic CRUD 页面已经存在，支持编辑 `input_uris` 和 `aggregator_config`
- Topic 聚合执行结果已经接入 observability
- 系统运行状态页面已经能够区分 `recipe` 和 `topic` 资源

当前缺口：

- Topic 管理页仍偏原型化，字段说明较少，容易误配
- 后台缺少针对 `input_uris` 的校验或预览能力
- 缺少 Topic 级详情页，无法直观看到输入源、最近错误、最近执行结果
- 聚合步骤的编辑器仍较原始，尚未产品化

## 4. 未来待办

### 4.1 优化 Topic 管理页文案和交互

- 明确区分：
  - 内部 Recipe URI
  - 内部 Topic URI
  - 外部 HTTP RSS URI
- 为聚合步骤提供更清晰的选项说明，而不是只暴露通用 key/value

### 4.2 增加配置校验与预览能力

- 在保存前校验 `input_uris` 的格式是否合法
- 校验 `aggregator_config` 是否完整
- 未来可考虑增加“预览 Topic 执行结果”按钮

### 4.3 增加 Topic 运行详情视图

- 展示：
  - 当前健康状态
  - 最近执行记录
  - 最近失败的上游 provider
  - 当前输入源配置

### 4.4 补齐可观测联动

- 将 Topic 的部分失败信息在后台界面中展示得更直观
- 区分：
  - 上游部分失败
  - 全部失败
  - 成功但结果为空

## 5. 追踪建议

这个领域完成的标准是：

- Topic 可以被稳定配置，而不是依赖使用者记忆内部约定
- Topic 异常可以在后台直接看出来，而不是只能翻日志
- 管理体验和运行状态可以支撑长期维护，而不只是 MVP 演示
